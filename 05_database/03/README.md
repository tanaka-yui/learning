# 第3章: DB設計応用 — キャッシュ・事前集計・非同期処理

## この章で学ぶこと

- 投稿時にフォロワーのタイムラインを事前計算する **Fan-out on Write** パターン
- Valkey/Redis を使ってタイムラインをメモリ上で管理する **インメモリタイムライン** パターン
- JOINクエリを維持しつつ、参照データを共有メモリにキャッシュする **共有キャッシュ** パターン

## 第2章からの接続

第2章の[非正規化パターン](../02/docs/04_denormalization.md)では、集計キャッシュカラムや冗長外部キーなど **DBレベル** の非正規化を学んだ。これらはSQLだけで完結する手軽なアプローチだが、スケールが大きくなると限界がある。

本章では、その次のステップとして **アプリケーションレベル** のキャッシュ・事前集計・非同期処理を扱う。DBだけでなくキューやインメモリストアを組み合わせることで、大規模サービスの読み取り性能を向上させる設計パターンを学ぶ。

## SNSアプリケーションについて

本章では第2章と同じSNSアプリケーションのスキーマを使用する。スキーマの詳細は[第2章 スキーマ定義](../02/docs/00_schema.md)を参照。

主に使用するテーブル:

- `users` — ユーザー
- `posts` — 投稿
- `follows` — フォロー関係
- `hashtags` / `hashtag_posts` — ハッシュタグ
- `hashtag_follows` — ハッシュタグフォロー

## 3つのパターンの比較

| 観点 | Fan-out on Write | インメモリタイムライン | 共有キャッシュ |
|------|-----------------|---------------------|--------------|
| データ格納先 | RDB（`timeline_entries`） | Valkey/Redis Sorted Set | RDB + Valkey/Redis |
| 読み取りコスト | 低（事前計算済み） | 最低（メモリ直読み） | 中（JOIN + キャッシュ） |
| 書き込みコスト | 高（フォロワー数に比例） | 高（フォロワー数に比例） | 低（通常のINSERT） |
| 整合性 | 結果整合性 | 結果整合性 | キャッシュ戦略に依存 |
| 障害時の影響 | キュー遅延 → タイムライン更新遅延 | メモリ消失 → 再構築が必要 | キャッシュミス → DB負荷増大 |
| 適用場面 | 読み取り多・フォロワー数が中規模 | 超低レイテンシが必要 | 既存JOINクエリを維持したい |
| インフラ追加 | メッセージキュー | Valkey/Redis | Valkey/Redis |
