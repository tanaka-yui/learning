# インデックス

## 1. インデックスとは

インデックスとは、データベースが特定の行を**高速に見つけ出すための補助データ構造**です。

書籍の索引（インデックス）に例えると分かりやすいです。

| 状況 | 本の場合 | データベースの場合 |
|------|---------|-----------------|
| インデックスなし | 先頭から全ページをめくって探す | テーブル全行を順に読む（Seq Scan） |
| インデックスあり | 索引で該当ページを確認して直接開く | インデックスで行の場所を特定して直接アクセス（Index Scan） |

インデックスがない状態でも正しい結果は返ってきますが、テーブルが大きくなるほど**検索のコストが線形に増加**します。適切なインデックスを作成することで、数万〜数十万件のデータを扱うクエリを大幅に高速化できます。

---

## 2. B-treeインデックスの仕組み

PostgreSQL のデフォルトインデックスは **B-tree（バランス木）** です。

### 特徴

- **バランス木構造**: どのキー値に対しても、根（ルート）から葉（リーフ）までの深さが一定
- **O(log N) アクセス**: データ件数が 100 万件でも、約 20 ステップで目的の行を見つけられる
- **葉ノードにソート済みのキー値とテーブル行へのポインタを格納**: 実際のテーブル行（ヒープ）への参照（CTID）を持つ
- **範囲検索も有効**: 葉ノードが順序を保っているため、`BETWEEN`・`>`・`<` も効率的に処理できる

### 概念図

```
              [50]
             /    \
         [25]      [75]
        /    \    /    \
  [1..24][25..49][50..74][75..100]
```

- 検索値が 30 の場合: ルート `[50]` → 左の `[25]` → `[25..49]` ノードへ（3ステップ）
- 検索値が 80 の場合: ルート `[50]` → 右の `[75]` → `[75..100]` ノードへ（3ステップ）

木が深くなっても、各ノードが多くの子を持てる（B木の特性）ため、実際の深さは非常に浅く抑えられます。

## 3. インデックスの種類

### 単一カラムインデックス

最も基本的な形。特定の1カラムを検索キーとするインデックス。

```sql
CREATE INDEX idx_posts_user_id ON posts(user_id);
```

`WHERE user_id = 42` のような等値検索や、`ORDER BY user_id` に効果的です。

### 複合インデックス（左端プレフィックスルール）

複数のカラムを組み合わせたインデックス。**左端のカラムから順に使われる**という重要なルールがあります。

```sql
CREATE INDEX idx_follows_user ON follows(user_id, follow_user_id);
```

| 条件 | インデックス使用 |
|------|--------------|
| `WHERE user_id = 1` | ✅ 使われる |
| `WHERE user_id = 1 AND follow_user_id = 42` | ✅ 使われる |
| `WHERE follow_user_id = 42` | ❌ 使われない（左端でないため） |

`follow_user_id` だけで検索したい場合は、`follow_user_id` を先頭にした別インデックスが必要です。

### 部分インデックス（条件付きインデックス）

特定の条件を満たす行だけをインデックス化します。

```sql
-- 例: 未削除の投稿だけをインデックス化（deleted_at カラムがある場合）
CREATE INDEX idx_posts_active ON posts(user_id) WHERE deleted_at IS NULL;
```

**有効な場面**: テーブルの一部の行だけが頻繁にアクセスされる場合（例: ステータスが `active` のレコードのみ）。インデックスのサイズを小さく保てるため、更新コストを下げつつ検索を高速化できます。

### カバリングインデックス（INCLUDE）

インデックスに検索キー以外のカラムも含める（カバー）ことができます。

```sql
CREATE INDEX idx_posts_user_covering ON posts(user_id) INCLUDE (id, created_at);
```

`SELECT` で必要なカラムがすべてインデックスに含まれていれば、PostgreSQL は **Index Only Scan** を実行します。テーブル本体（ヒープ）へのアクセスが不要になるため、さらなる高速化が期待できます。

## 4. どのカラムにインデックスを張るか

インデックスを張るかどうかの判断基準をまとめます。

| カラムの特性 | 指針 | 理由 |
|------------|------|------|
| 外部キー（FK）カラム | 必ず張る | JOIN や参照整合性チェックで頻繁に使われる |
| `WHERE` で頻繁に使うカラム | 張る | 検索の絞り込みに直接効く |
| `ORDER BY` で頻繁に使うカラム | 張る | ソート処理をインデックスで代替できる |
| 高選択性（値がほぼユニーク）なカラム | 効果大 | 少数の行に絞り込めるためインデックス効果が高い |
| 低選択性（boolean やステータスなど）なカラム | 効果薄 | 多数の行がヒットするため Seq Scan と差が出にくい |

> **選択性（Selectivity）**: カラムに含まれるユニークな値の割合。値の種類が多いほど選択性が高い。

## 5. インデックスが逆効果なケース

インデックスは万能ではありません。以下のケースでは効果がないか、むしろパフォーマンスを下げることがあります。

### テーブルが小さい（数百行程度）

Seq Scan の方がインデックスを経由するよりも速い場合があります。PostgreSQL のクエリプランナーはコストを見積もって自動的に Seq Scan を選択します。

### 書き込みが非常に多いテーブル

`INSERT` / `UPDATE` / `DELETE` のたびにインデックスも更新されます。インデックスが多いほど書き込みコストが増加し、書き込みスループットが低下します。

### 低選択性カラム（例: boolean）

```sql
-- active が TRUE / FALSE しかない場合
CREATE INDEX idx_posts_active ON posts(active);  -- 効果薄
```

`active = TRUE` の行が全体の 80% を占めるような場合、インデックスで絞り込んでもほぼすべての行にアクセスすることになり、Seq Scan と変わりません。

## 6. 実践例 — follows テーブルの複合インデックス

「演習ユーザー（`:'my_id'`）がフォローしているユーザーの最新投稿を取得する」というタイムライン系クエリを例に、インデックスの効果を確認します。

### Before: follows.user_id にインデックスがない状態

```
Hash Join  (cost=3135.00..34731.00 rows=2000 width=...)
  Hash Cond: (p.user_id = f.follow_user_id)
  -> Seq Scan on posts  (cost=0.00..23310.00 rows=1000000 ...)
  -> Hash  (cost=3085.00..3085.00 rows=4000 ...)
       -> Seq Scan on follows  (cost=0.00..3085.00 rows=4000 ...)
            Filter: (user_id = :'my_id')
Execution Time: 145.2 ms
```

**問題点:**
- `follows` テーブルの全 300,000 件を Seq Scan してから `user_id = :'my_id'` の行を絞り込んでいる
- `posts` テーブルの全 1,000,000 件も Seq Scan して Hash Join している
- 結合前に双方の大きなテーブルをフルスキャンしているため時間がかかる

### After: 複合インデックス `idx_follows_user` を追加した状態

```
Nested Loop  (cost=0.56..845.20 rows=2000 width=...)
  -> Index Scan using idx_follows_user on follows  (cost=0.42..54.30 rows=30 ...)
       Index Cond: (user_id = :'my_id')
  -> Index Scan using posts_pkey on posts  (cost=0.56..15.30 rows=1 ...)
       Index Cond: (user_id = f.follow_user_id)
Execution Time: 1.8 ms
```

**改善ポイント:**
- `follows` テーブルへのアクセスが Seq Scan から **Index Scan** に変わり、該当ユーザーのフォロー先（約 30 件）だけを取得できるようになった
- 取得した 30 件の `follow_user_id` に対して `posts_pkey` を使って Nested Loop で結合するため、`posts` のフルスキャンも不要になった
- 実行時間が **145.2 ms → 1.8 ms**（約 80 倍高速化）

このように、結合の起点となるカラム（外部キーや絞り込み条件に使うカラム）にインデックスを張ることで、クエリの実行計画が劇的に改善されます。
