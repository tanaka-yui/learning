# SQLインジェクション

## 1. 概要

### SQLインジェクションとは何か

SQLインジェクションとは、アプリケーションがユーザーからの入力を適切に処理せずにSQL文を組み立てる際に、攻撃者が意図的なSQL断片を入力として注入（インジェクション）し、本来想定されていないSQL文を実行させる攻撃手法である。

### なぜ危険か

SQLインジェクションが成功すると、以下のような深刻な被害が発生する。

- **データ漏洩**: 本来アクセスできないはずのデータ（個人情報、パスワード、機密情報）が取得される
- **データ改ざん**: UPDATE文やINSERT文を注入され、データが不正に変更・追加される
- **認証バイパス**: ログイン処理のSQL文を操作し、パスワードなしで認証を突破される
- **DB破壊**: DROP TABLEやDELETE文を注入され、データベースが破壊される

### OWASP Top 10 での位置づけ

SQLインジェクションは、OWASP Top 10において「A03:2021 - Injection（インジェクション）」に分類されている。インジェクション攻撃は長年にわたりWebアプリケーションにおける最も深刻な脅威の一つとして認識されており、2017年版では第1位に位置づけられていた。2021年版でも引き続きトップ3に含まれている。

## 2. 攻撃の仕組み

### 文字列結合によるSQL構築の危険性

アプリケーションがユーザー入力を文字列結合でSQL文に埋め込む場合、攻撃者はSQL構文の一部として解釈される文字列を入力することで、クエリの意味を改変できる。

### ユーザー入力がSQLクエリに組み込まれる流れ

**正常な場合:**

```
ユーザー入力: Alice
生成されるSQL: SELECT id, name, email FROM users WHERE name = 'Alice'
結果: Aliceのレコードのみ返される
```

**攻撃の場合:**

```
ユーザー入力: ' OR '1'='1
生成されるSQL: SELECT id, name, email FROM users WHERE name = '' OR '1'='1'
結果: WHERE句が常にTRUEになり、全レコードが返される
```

図解すると以下のようになる。

```
通常のフロー:
  [ユーザー入力: Alice]
       |
       v
  fmt.Sprintf("SELECT ... WHERE name = '%s'", "Alice")
       |
       v
  SELECT ... WHERE name = 'Alice'
       |
       v
  [Aliceのレコードのみ返却]

攻撃のフロー:
  [ユーザー入力: ' OR '1'='1]
       |
       v
  fmt.Sprintf("SELECT ... WHERE name = '%s'", "' OR '1'='1")
       |
       v
  SELECT ... WHERE name = '' OR '1'='1'
       |                        ^^^^^^^^
       |                        常にTRUE
       v
  [全レコードが返却される]
```

## 4. 攻撃手順（ステップバイステップ）

### 攻撃1: 全ユーザー取得

1. 脆弱版 (http://localhost:3000) をブラウザで開く
2. 検索フォームに以下を入力する
   ```
   ' OR '1'='1
   ```
3. 全ユーザー（Alice, Bob, Charlie）が表示される。本来は入力した名前に一致するユーザーのみが返されるべきだが、WHERE句が常にTRUEとなるため全レコードが漏洩する

### 攻撃2: テーブル構造の取得

1. 脆弱版 (http://localhost:3000) の検索フォームに以下を入力する
   ```
   ' UNION SELECT 1, sql, 3 FROM sqlite_master--
   ```
2. テーブルのCREATE文が表示される。これによりDB構造（テーブル名、カラム名、データ型）が攻撃者に漏洩する
3. この情報を元に、攻撃者はさらに精度の高い攻撃（パスワードカラムの抽出など）を実行できるようになる

### 攻撃3: 対策版で同じ攻撃を試す

1. 対策版 (http://localhost:3001) をブラウザで開く
2. 攻撃1と同じペイロード `' OR '1'='1` を入力する
3. 結果が0件になることを確認する。プリペアドステートメントにより、入力値がSQL構文として解釈されず、文字列 `' OR '1'='1` という名前のユーザーを検索するだけとなる

## 5. 脆弱なコードの解説

ファイル: `sql-injection/vulnerable/backend/main.go`

脆弱版のコアとなる処理は以下の部分である。

```go
name := r.URL.Query().Get("name")

var query string
if name != "" {
    // 意図的に脆弱なSQL: 文字列結合によるSQLインジェクション脆弱性
    // 教育目的のため、プリペアドステートメントを使用していない
    query = fmt.Sprintf("SELECT id, name, email FROM users WHERE name = '%s'", name)
} else {
    query = "SELECT id, name, email FROM users"
}

rows, err := db.Query(query)
```

### なぜ危険か

`fmt.Sprintf` を使用してユーザー入力（`name`）をそのままSQL文に埋め込んでいる。この方法では、ユーザー入力に含まれるシングルクォートやSQL構文がそのままSQL文の一部として解釈される。データベースドライバはこの文字列を「完成されたSQL文」として受け取るため、入力値とSQL構文の区別ができない。

## 6. 対策コードの解説

ファイル: `sql-injection/secure/backend/main.go`

対策版のコアとなる処理は以下の部分である。

```go
name := r.URL.Query().Get("name")

var rows *sql.Rows
var err error

if name != "" {
    // プリペアドステートメントを使用してSQLインジェクションを防止する
    // プレースホルダー(?)を使うことで、ユーザー入力がSQL文として解釈されなくなる
    rows, err = db.Query("SELECT id, name, email FROM users WHERE name = ?", name)
} else {
    rows, err = db.Query("SELECT id, name, email FROM users")
}
```

### プリペアドステートメントの仕組み

プリペアドステートメント（パラメータ化クエリ）では、以下の3つのステップによりSQLインジェクションを防止する。

1. **SQL構文とデータの分離**: SQL文のテンプレート（`WHERE name = ?`）とユーザー入力（`name`の値）が別々にデータベースエンジンへ送信される。SQL文の構文解析はユーザー入力を受け取る前に完了しているため、入力値がSQL構文を変更することは不可能である
2. **データベースドライバによるエスケープ処理**: プレースホルダー `?` に渡される値は、データベースドライバが自動的に適切なエスケープ処理を行う。開発者が手動でエスケープする必要がない
3. **ユーザー入力がSQL構文として解釈されない**: `' OR '1'='1` のような入力が渡されても、それは純粋な「文字列データ」として扱われる。シングルクォートやSQL予約語が構文的な意味を持つことはない

---

## 7. まとめ -- ベストプラクティスチェックリスト

- [ ] **プリペアドステートメント（パラメータ化クエリ）を常に使用する**: SQL文を組み立てる際は必ずプレースホルダーを使い、ユーザー入力を直接埋め込まない
- [ ] **ORMの利用を検討する**: GORM等のORMを利用することでSQLを直接書く機会を減らし、インジェクションのリスクを低減する
- [ ] **入力バリデーション（許可リスト方式）を実施する**: ユーザー入力に対して、許可された文字・パターンのみを受け入れる。拒否リスト方式（危険な文字を除外する方式）は漏れが生じやすいため推奨しない
- [ ] **最小権限の原則を適用する**: データベースユーザーに必要最小限の権限のみを付与する。Webアプリケーション用のDBユーザーにDROPやALTER権限を付与しない
- [ ] **エラーメッセージにDB情報を含めない**: データベースのエラーメッセージをそのままユーザーに返さない。攻撃者にテーブル構造やカラム名などのヒントを与えることになる
